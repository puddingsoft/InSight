# InSight
见智投研Agent

一、先泼一盆冷水：这类产品最容易踩的坑
股票投资垂直智能体，难点 90% 不在对话界面，而在背后的"数据 + 分析逻辑 + 合规"。我见过太多类似项目，UI 做得很漂亮，但一问"为什么推荐这只票"就露馅。所以方案要先解决三个根本问题：

数据从哪来：行情、财报、公告、新闻——没有可靠数据源，智能体就是在"一本正经地胡说八道"。
分析逻辑可解释：用户敢不敢信，取决于结论能不能溯源到具体数据，而不是大模型凭感觉编。
合规红线：在中国境内，向不特定公众提供具体个股买卖建议属于"证券投资咨询业务"，需要牌照（证监会的投资咨询资格）。这条必须想清楚，直接决定产品形态。
关于第 3 点，我想先和你确认方向，这会影响整个架构设计。

明确了：投研辅助工具定位（不给明确买卖指令，合规友好）+ 完整可上线产品 + 全 JS/TS 技术栈。这是一个很务实的组合。下面给你完整方案。

二、产品形态：一个对话框 + 三个"视图"
很多人会把"每日复盘 / 基本面分析 / 操作决策"做成三个割裂的页面。更好的做法是：它们是同一个对话智能体在不同场景下的三种工作模式（Agent 的三种"人格/技能"），共享同一套数据底座和记忆。

页面	本质	智能体在做什么	关键产出
每日复盘	定时/触发任务	拉当日行情+持仓+新闻，总结涨跌归因	复盘报告卡片
基本面分析	检索+推理	拉财报/估值/行业数据，结构化分析	财务诊断 + 估值雷达
操作决策	多信号汇总	综合技术面+基本面+情绪面给"参考视角"	决策辅助看板（非指令）
注意"操作决策"这个名字在投研辅助定位下，对外措辞建议改成 "决策参考 / 持仓体检"，输出"需要关注的因素"而非"该买该卖"，这样合规风险最低。

四、技术选型（全 JS/TS）
层	选型	理由
全栈框架	Next.js 14 (App Router)	前后端一体，API Routes 直接写后端，省去前后端分离的协调成本
对话 UI	Vercel AI SDK (ai 包)	useChat + 流式输出 + Tool Calling 几乎开箱即用，是这类产品的最优解
组件库	shadcn/ui + Tailwind	富卡片、图表容器好搭
图表	ECharts / Lightweight-charts	K线用 TradingView 的 lightweight-charts，财务图表用 ECharts
LLM	DeepSeek / 通义千问（国内）或 GPT-4o	国内合规优先用国产模型，支持 Function Calling
数据库	Postgres (Supabase) + Drizzle ORM	存用户、持仓、对话历史、复盘记录
缓存/队列	Redis + BullMQ	行情缓存（避免重复打 API）、每日复盘定时任务
数据源	行情：Tushare / AKShare(需 Python 微服务) / 新浪财经接口；财报：Tushare	详见下文
一个现实提醒：A股最干净的开源数据是 Tushare 和 AKShare，但它们是 Python 库。在全 TS 栈里，要么调它们的 HTTP 接口（Tushare 有 REST），要么单独起一个轻量 Python 数据微服务专门取数。这是全 TS 栈唯一需要妥协的地方。

五、对话驱动三视图的交互逻辑
这是体验的灵魂——用户说话，智能体判断意图，调用工具，把结构化结果渲染成对应页面的卡片：

用户: "今天我的持仓怎么样"
  → 意图识别: 复盘
  → 调用 getPortfolio() + getQuotes() + getNews()
  → LLM 总结归因
  → 前端渲染「每日复盘卡片」（涨跌榜 + 归因 + 新闻）

用户: "贵州茅台基本面如何"
  → 意图识别: 基本面
  → 调用 getFinancials("600519") + calcValuation()
  → 前端渲染「财务诊断 + 估值雷达图」

用户: "这只票现在适合关注吗"
  → 意图识别: 决策参考
  → 汇总技术面+基本面+情绪面信号
  → 前端渲染「多因子打分看板」(强调"参考，非建议")
关键设计点：LLM 的工具返回值是结构化 JSON，前端按 type 字段决定渲染哪种富卡片。这就是"对话即页面"的实现方式。

七、应该如何"开始"——具体第一步
落到执行层面，第一步不要碰真实数据，先用 Mock 把"对话 → 工具 → 富卡片"这条链路打通。验证体验跑得通了，再往里灌真数据。建议这样起步：

初始化项目：npx create-next-app + Tailwind + shadcn/ui，装 ai 和 @ai-sdk/openai（或 deepseek）。
搭对话壳：用 AI SDK 的 useChat 做出流式聊天框，左侧/顶部放"每日复盘 / 基本面 / 决策参考"三个 Tab。
定义 2~3 个 Mock 工具：如 getQuote、getFinancials，先返回写死的 JSON，让 LLM 学会调用它们。
做富卡片渲染：根据工具返回 JSON 的 type 渲染对应卡片（行情卡、财务卡、雷达图）。
这一步跑通后，再进入阶段 2 替换成真实数据源。
八、必须记住的三条红线
合规话术层：所有涉及操作的输出，统一过一层后处理，把"建议买入"改成"以下因素值得关注"，并强制附带风险免责声明。这是投研辅助定位的生命线。
数据可溯源：每个结论后面带数据来源和时间戳，否则 LLM 会"幻觉"出不存在的财报数字。
不存储不该存的东西：用户持仓属敏感信息，加密存储、明确隐私协议。
这份方案就是你"如何开始"的完整蓝图：先想清合规定位 → 全 TS 架构 → 对话驱动三视图 → 四阶段从 Mock 到上线
